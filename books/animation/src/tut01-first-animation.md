# A Clip Plays

By the end of this chapter a hero breathes and a training dummy stands menacingly still-ish,
both animated from a real sprite-sheet export, and you'll know every piece of vocabulary the
rest of the book builds on: sheets, regions, tags, clips, and players.

## Step 0 — create your project

From the repository root, the same move as every Fulcrum code-along — the workspace's
`games/*` glob picks the new crate up automatically:

```text
cargo new games/my-dojo
```

Replace `games/my-dojo/Cargo.toml`:

```toml
[package]
name = "my-dojo"
version = "0.1.0"
edition = "2024"

[dependencies]
fulcrum = { workspace = true }
# Manifest-only: lets #[derive(Component)] resolve the `bevy_ecs::` paths it generates.
bevy_ecs = { workspace = true }
```

And borrow the finished game's art — sprite sheets, their JSON descriptions, the state
machine files you'll use from chapter 3, and the white pixel for the floor:

```text
mkdir games/my-dojo/assets
cp games/dojo/assets/hero.png games/dojo/assets/hero.json \
   games/dojo/assets/dummy.png games/dojo/assets/dummy.json \
   games/dojo/assets/white.png games/my-dojo/assets/
cp -r games/dojo/assets/anim games/my-dojo/assets/anim
```

## What's actually in those files

Open `assets/hero.json` before writing any code — the whole animation system is easier to
believe once you've seen that it's just data. It's an **Aseprite export**: the de-facto
standard interchange for pixel art, produced by

```text
aseprite -b hero.ase --sheet hero.png --data hero.json --format json-array --list-tags
```

(Ours was generated by `tools/gen_dojo_art.py` instead — same format, no artist harmed. The
format is plain enough to hand-write, which the reference chapter covers.) Two halves matter:

```json
{ "filename": "hero attack 12", "frame": {"x":64,"y":16,"w":16,"h":16}, "duration": 70 },
```

Each **frame** entry names a rectangle in `hero.png` — a **region** — and how long it should
show, in milliseconds. The packed PNG is just frames side by side; the JSON is the map.

```json
{ "name": "attack", "from": 10, "to": 13, "direction": "forward", "repeat": "1" }
```

Each **tag** names a range of frames: this one says frames 10–13 are, together, the attack
animation, played once (`"repeat": "1"` — loop-forever tags omit the field). Aseprite writes
tags when the artist labels a timeline range; they're about to become Fulcrum clips.

## Step 1 — load it and spawn something animated

`src/main.rs`, complete:

```rust,ignore
use fulcrum::prelude::*;

fn setup(mut commands: Commands, mut aseprite: AsepriteLoader, mut camera: ResMut<Camera2D>) {
    camera.scaling = ScalingMode::Letterbox {
        width: 320.0,
        height: 180.0,
    };
    camera.center = vec2(160.0, 90.0);

    let hero = aseprite.load("hero.json").expect("hero sheet loads");
    commands.spawn((
        Sprite::from_sheet(hero.sheet, 0).with_z(1.0),
        Transform2D::from_xy(120.0, 90.0),
        AnimationPlayer::play(hero.clips["idle"]),
    ));

    let dummy = aseprite.load("dummy.json").expect("dummy sheet loads");
    commands.spawn((
        Sprite::from_sheet(dummy.sheet, 0).with_z(1.0),
        Transform2D::from_xy(200.0, 90.0),
        AnimationPlayer::play(dummy.clips["idle"]),
    ));
}

fn main() {
    Fulcrum::new("my-dojo: a clip plays")
        .insert_resource(AssetServer::new(concat!(env!("CARGO_MANIFEST_DIR"), "/assets")))
        .with_plugin(DefaultPlugins)
        .add_startup(setup)
        .run();
}
```

```text
cargo run -p my-dojo
```

The hero breathes — a slow four-frame bob. The dummy blinks between its two idle frames.
Nobody wrote an update loop for either. Unpack the three new tools:

> **Toolbox — `AsepriteLoader`:** one call turns an export into everything usable: a
> `SpriteSheet` asset (the PNG plus every named region), and one **`AnimationClip`** per
> tag, returned as `clips` keyed by tag name. Handles all around — load once, copy the
> tickets freely.
>
> **Toolbox — `AnimationClip`:** a sequence of sheet regions plus a duration for each. One
> conversion happened silently at load, and it's the most important line in this chapter:
> the file's *milliseconds* became simulation *ticks* (100 ms → 6 ticks at 60 Hz). From
> here on, animation time is tick time — the same clock the simulation runs on, not the
> wall clock the renderer runs on. Chapter 4 spends this; chapter 5 proves it.
>
> **Toolbox — `AnimationPlayer`:** a component: "this entity is playing that clip." Every
> tick, an engine system advances every player and writes the current frame's region into
> the entity's `Sprite`. That's the whole runtime — you never advance anything by hand.
> One ownership rule to tattoo somewhere: **while a player is attached, it owns
> `Sprite::region`.** Set the region manually and the player will overwrite it next tick.

Note what `Sprite::from_sheet(hero.sheet, 0)` contributes: the initial frame, and the
sprite's size (from the region). The player takes over from the first tick.

## Step 2 — prove it's data

Two experiments before moving on, both zero-code:

1. Open `hero.json`, change the idle frames' `duration` from 160 to 400, rerun. The breath
   slows. Animation timing is content, not code — which is why the artist can own it.
2. Change `AnimationPlayer::play(hero.clips["idle"])` to `hero.clips["run"]`. The hero jogs
   in place. A clip is a value; playing a different one is passing a different value.

## Checkpoint

Your program should match `games/dojo/examples/an01_clip.rs`:

```text
cargo run -p dojo --example an01_clip
```

New vocabulary:

| Term | What it is |
| --- | --- |
| sheet / region | one packed PNG / a named rectangle in it |
| tag | an artist-labeled frame range in the export |
| `AnimationClip` | a tag, imported: regions + per-frame durations **in ticks** |
| `AnimationPlayer` | component that plays a clip on an entity's `Sprite` |
| `AsepriteLoader` | export in, sheet + clips out |

## Exercises

1. **A crowd.** Spawn five heroes at different positions, all playing `idle`. Notice they
   breathe in eerie unison — every player started on the same tick. Now stagger them: give
   each spawned `AnimationPlayer` a different starting `frame_index` (it's a public field).
   Crowd desynchronization is a one-line trick every game reuses.
2. **Read the export.** Without running anything, work out from `hero.json` alone: how many
   ticks does one full run cycle take at 60 Hz? Then verify your arithmetic — spawn a hero
   playing `run` next to one playing `idle` and count bobs against breaths.
3. **Pingpong (harder).** Add a new tag to `dummy.json` by hand: `"sway"`, covering the two
   idle frames with `"direction": "pingpong"`. Play it. Then check the reference chapter's
   importer section to see exactly what pingpong expanded to, and why the ends aren't
   doubled.

The hero breathes but can't do anything else. Next chapter adds movement and a sword — and
switches clips the obvious way, so you can feel exactly why chapter 3 exists.
